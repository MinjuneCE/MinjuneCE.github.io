행안부 중앙영구기록관리시스템 기능개선

2024 기능개선 핵심 프로젝트
상황

서버 1대씩 고정 배정된 기존 구조를 8개 서버 HA 구성으로 전환해야 했습니다. 그런데 L4 스위치 도입은 불가했습니다. 하드웨어 없이, 순수하게 소프트웨어만으로 분산 처리를 설계해야 하는 상황. "이게 가능한가?"라는 의문에서 프로젝트가 시작되었습니다.

맡은 역할: 다중 서버 분산 아키텍처 설계 / 동시성 제어 / SQL 성능 튜닝

Problem 1 — 분산 설계의 딜레마

Master-Slave 방식을 적용하면 서버 역할이 고정됩니다. 하지만 현장 요구사항은 인수 3 : 품질 5에서 인수 5 : 품질 3으로 상황에 따라 유동적으로 배분하는 것이었습니다. 고정 역할 배분으로는 절대 풀 수 없는 구조적 문제였습니다.

Solution — SELECT FOR UPDATE 기반 자율 할당

"각 서버가 스스로 작업을 가져가면 어떨까?"라는 아이디어에서 출발했습니다. Oracle 스케줄링 테이블에 작업 대기열을 관리하고, 각 서버가 SELECT FOR UPDATE SKIP LOCKED로 미처리 작업을 직접 선점하는 구조를 설계했습니다. Master 없이 수평 확장이 가능하고, 트랜잭션 레벨 락으로 중복을 원천 차단합니다.

Problem 2 — Oracle Deadlock

분산 구조를 적용하자 새로운 문제가 나타났습니다. 여러 서버가 동시에 같은 테이블을 잠그면서 Deadlock이 발생하고 데몬이 멈추는 상황이 반복되었습니다. 운영 중인 공공 시스템이라 장애 허용 범위가 극히 제한적이었습니다.

Solution

Oracle AWR 리포트와 v$lock 뷰를 분석해 교착 지점을 특정하고, 락 획득 순서를 일관되게 정립했습니다. 트랜잭션 예외 발생 시 데몬이 중단되지 않도록 안전한 복구 로직을 추가했습니다.

Problem 3 — SQL 성능 저하

A→B→C→D로 이어지는 다중 테이블 관계에서 EXISTS 중첩 쿼리의 응답 시간이 급격히 느려지는 문제가 있었습니다.

Solution

PK/FK 구조를 분석하고 Oracle 실행계획(EXPLAIN PLAN)을 기반으로 쿼리 접근 순서를 재설계했습니다. 인덱스 힌트와 조인 방식을 조정해 성능을 개선했습니다.

맡은 역할: 운영 환경 복합 장애 원인 분석 및 근본적 해결

Problem

로그만으로는 원인을 특정할 수 없었습니다. 배치 처리, 데몬 상태, DB 커넥션, 외부 연동까지 여러 계층에 걸친 원인이 복합적으로 얽혀 있었고, 이전 담당자들도 해결하지 못한 채 넘어온 이슈들이었습니다.

내부 인프라 개선

내부 프로젝트 DevOps
상황

팀원 각자의 로컬 PC에 개발환경이 파편화되어 있었습니다. "내 PC에서는 되는데?"가 일상이었고, 빌드와 배포는 전부 수동이었습니다. 아무도 나서지 않는 문제였지만, 팀 생산성의 병목이 여기에 있다고 판단했습니다.

맡은 역할: Docker 기반 통합 환경 설계, Jenkins CI/CD 도입, 모니터링 구축

Problem

각 PC마다 Java 버전, DB 설정, 환경변수가 달랐고, 배포할 때마다 수동으로 빌드→복사→재시작을 반복하고 있었습니다.

Solution

OCI 개발 서버에 Docker 기반 통합 환경을 구축하고, Jenkins 파이프라인을 설계해 빌드·배포를 자동화했습니다. Prometheus + Grafana 기반 모니터링 체계도 직접 구축했습니다.
